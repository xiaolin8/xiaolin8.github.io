Nginx必知必会

Posted on 2018-08-12 |

简介

  • 高性能
  • 优秀的架构（无需重构，长期有效）
  • 高可靠
  • 热部署

应用场景

  • 静态资源服务
  • 反向代理服务（缓存加速、负载均衡）
  • API 服务（OpenResty）

基础配置

  • $符号代表变量
  • nginx -s quit 优雅重启

  • nginx -s reopen 优雅重启

  • nginx包含一个master进程和多个worker进程，如果开启缓存的话，还会启动缓存加载以及缓存管理的进程

  • work进程数目的设置可以在nginx.conf指定，也可以根据cpu数目调整(worker_processes auto)

六大区域：
main(全局设置)、events(nginx工作模式)、http(http设置)、
sever(主机设置)、location(URL匹配)、upstream(负载均衡服务器设置)

web server

Server

Server的配置在http模块里面，通过listen port以及server_name识别具体http服务

Notice:

  • 如果只允许本机访问，可以设置 listen localhost:80
  • 默认default server就是在nginx.conf文件里面第一个server的配置
  • server_name 匹配多个的时候使用空格分离；也可以支持匹配符，比如 .example.org, example.;

location

正则表达式以~表示大小写区分的匹配， ~* 表示大小写不区分的匹配。
location ~ .html? { //匹配包含.html或者.htm的请求URIs
…
}

  • = 精确匹配
  • 如果配置中包含 ^~, 则所有的正则表达式都不匹配
  • Nginx匹配是取先匹配到的正则表达式，所以最经常访问的请求放在前面，可以加速匹配速度

return

return可以使用在location或者server模块中，用于返回一个错误码或者重定向。

rewrite

rewrite可以使用在location或者server模块中，可以重写URI，来匹配到最合适的处理。
它包含两个必需参数和一个可选参数。第一个必需参数是可匹配到URI的正则表达式， 第二个必需参数是用于替换该URI的URI。一个可选参数是一个标识，决定后续处理逻辑。
可选参数有：

  • last: 停止当前server或者location模块内rewrite执行。如果其他location也有匹配到该URI的rewrite, 也可以执行。
  • break：停止当前server或者location模块内rewrite执行，并取消其他location针对该URI的检索。

error_page

error_page 可以处理错误逻辑，返回正确的页面
error_page 404 /404.html;

静态资源处理

root

root指定一个文件目录，该设置可以在http, server, location中。

index

index指定目录中默认显示文件，如果显示文件列表，则设置 autoindex on;

try_files

try_files用于检测指定的文件或者目录是否存在，如果不存在就做一个内部跳转

反向代理

反向代理方式实际上就是一台负责转发的代理服务器，貌似充当了真正服务器的功能，但实际上并不是，代理服务器只是充当了转发的作用，并且从真正的服务器那里取得返回的数据。这样说，其实nginx完成的就是这样的工作。我们让nginx监听一个端口，譬如80端口，但实际上我们转发给在8080端口的tomcat，由它来处理真正的请求，当请求完成后，tomcat返回，但数据此时没直接返回，而是直接给nginx，由nginx进行返回，这里，我们会以为是nginx进行了处理，但实际上进行处理的是tomcat。

反向代理支持的后端可以是http server，也可以是非http server，比如FastCGI， uwsgi， SCGI， memcached。

处理请求头部

默认情况下，nginx会把转发的请求Host设置为$proxy_host, connection设置为close。
可以通过设置来修改其他的头部信息

配置缓存

默认情况下，nginx会缓存来自后端服务器的响应，直到所有响应包收到之后再发给请求方。 如果要禁止缓存，设置proxy_buffering off; 缓存大小也可以设置

proxy_buffer_size设置响应头的缓存大小，一般比数据要小；proxy_buffers设置缓存响应数据的个数以及大小。

压缩

压缩响应包可以减少传输数据的大小，但是会影响nginx的性能。 开启压缩：
gzip on;
gzip_types text/plain application/xml;
gzip_proxied no-cache no-store private expired auth;
gzip_min_length 1k; //default is 20 bytes
gzip_buffers 4 16k;
gzip_comp_level 2;
gzip_types text/plain application/x-javascript application/javascript text/css application/xml text/javascript application/x-httpd-php image/jpeg image/gif image/png;
gzip_vary off;

缓存

默认nginx会缓存所有的GET, HEAD请求
proxy_cache_key “$host$request_uri$cookie_user”; //改变缓存key
proxy_cache_min_uses 5; // 被调用5次之后才会被缓存
proxy_cache_methods GET HEAD POST;
proxy_cache_valid 200 302 10m; // 200, 302返回码10分钟有效
proxy_cache_valid 404 1m; // 404 返回码1分钟有效
proxy_cache_valid any 5m; //所有请求5分钟有效
proxy_cache_bypass $cookie_nocache $arg_nocache$arg_comment; // 如果其中一个变量不为空，就绕过缓存，直接转发至后端服务器
proxy_no_cache $http_pragma $http_authorization; // 如果其中一个变量不为空，则不缓存该URI

  2. 127.0.0.1 www.helloworld.com

负载均衡配置实战

假设这样一个应用场景：将应用部署在 192.168.1.11:80、192.168.1.12:80、192.168.1.13:80 三台 linux 环境的服务器上。网站域名叫 www.helloworld.com，公网 IP 为 192.168.1.11。在公网 IP 所在的服务器上部署 nginx，对所有请求做负载均衡处理。

多模块部署配置实战

举个例子：假如 www.helloworld.com 站点有好几个 webapp，finance（金融）、product（产品）、admin（用户中心）。访问这些应用的方式通过上下文(context)来进行区分:

www.helloworld.com/finance/

www.helloworld.com/product/

www.helloworld.com/admin/

我们知道，http 的默认端口号是 80，如果在一台服务器上同时启动这 3 个 webapp 应用，都用 80 端口，肯定是不成的。所以，这三个应用需要分别绑定不同的端口号。

那么，问题来了，用户在实际访问 www.helloworld.com 站点时，访问不同 webapp，总不会还带着对应的端口号去访问吧。所以，你再次需要用到反向代理来做处理。

https 反向代理配置

  • HTTPS 的固定端口号是 443，不同于 HTTP 的 80 端口
  • SSL 标准需要引入安全证书，所以在 nginx.conf 中你需要指定证书和它对应的 key

静态站点配置实战

举例来说：如果所有的静态资源都放在了 /app/dist 目录下，我们只需要在 nginx.conf 中指定首页以及这个站点的 host 即可。

文件服务器搭建实战

  • 将 autoindex 开启可以显示目录，默认不开启。
  • 将 autoindex_exact_size 开启可以显示文件的大小。
  • 将 autoindex_localtime 开启可以显示文件的修改时间。
  • root 用来设置开放为文件服务的根路径。
  • charset 设置为 charset utf-8,gbk;，可以避免中文乱码问题（windows 服务器下设置后，依然乱码，本人暂时没有找到解决方法）。

跨域实战

web 领域开发中，经常采用前后端分离模式。这种模式下，前端和后端分别是独立的 web 应用程序，例如：后端是 Java 程序，前端是 React 或 Vue 应用。

各自独立的 web app 在互相访问时，势必存在跨域问题。解决跨域问题一般有两种思路：

  • CORS
    在后端服务器设置 HTTP 响应头，把你需要运行访问的域名加入加入 Access-Control-Allow-Origin 中。

  • jsonp
    把后端根据请求，构造 json 数据，并返回，前端用 jsonp 跨域。

需要说明的是，nginx 根据第一种思路，也提供了一种解决跨域的解决方案。

举例：www.helloworld.com 网站是由一个前端 app ，一个后端 app 组成的。前端端口号为 9000， 后端端口号为 8080。
